# Reference App

`reference/standard-service` is the canonical application shape in Plumego. Choose this page when the next question is shape rather than execution: you want to see how the reference service is assembled, what its layout demonstrates, and which conclusions are safe to carry into a real service.

If you have not run anything yet and simply want the shortest working path, start with [Getting Started](/docs/getting-started). If your question has already moved beyond canonical service shape and into repository boundaries, open [Architecture](/architecture) first, then continue into [Repository Boundaries](/docs/concepts/repo-control-plane) when the question becomes how Plumego classifies ownership inside the repository.

## What it demonstrates

- explicit bootstrap in `main.go`
- app-local wiring in `internal/app/app.go`
- explicit route registration in `internal/app/routes.go`
- transport-only handlers under `internal/handler`

## Why it matters

Plumego is intentionally opinionated about one obvious way to structure a service. The reference app is the place where that path is made concrete and reviewable.

## When to follow it closely

Use it when you are:

- starting a new service
- validating handler and route shape
- deciding where application-local code should live
- comparing your service skeleton against Plumego’s canonical layout

## When to leave runnable shape and switch to repository boundaries

Stay on this page while the question is still, “How is the canonical service assembled?” Switch out of it as soon as the real question becomes ownership instead of shape.

Move to [Architecture](/architecture) and [Repository Boundaries](/docs/concepts/repo-control-plane) when you need to answer questions like:

- which stable root or `x/*` family should own this change
- whether a package belongs to the default path or an extension family
- where shared repository rules, docs, specs, or task cards should take over from service-local layout
- whether copying the reference app structure would blur a repository boundary that should stay explicit

A good rule of thumb is simple: if you are still comparing files, stay here; if you are classifying ownership, leave this page.

## What not to infer from it

- it is a canonical reference, not a feature catalog
- it does not make every `x/*` package part of the default path
- it should not be treated as a license to blur stable-root boundaries
- it is not the page that decides repository ownership or release maturity
