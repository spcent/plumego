# Repository Boundaries

Plumego is not only a set of Go packages. It also maintains a repository control plane that tells humans and agents how to classify work before they start editing code.

If you have not yet studied the canonical runnable service shape, pause here and read [Reference App](/docs/reference-app) first. Come back once the question is no longer “How is the service assembled?” but “Which repository surface owns this change?”

## When this page should take over from Reference App

Read this page after Reference App when file layout is no longer enough and you need repository-level classification instead.

Typical signals include:

- you are deciding which stable root or `x/*` family should own a change
- you need to separate service-local layout from repository-wide rules
- you are choosing whether docs, specs, or task cards should become the next source of truth
- you are reviewing a change and the main risk is boundary drift rather than runnable shape

## The three surfaces

### Docs

Docs explain intent for humans. They describe the canonical path, module boundaries, release posture, and the default way to navigate the repository.

### Specs

Specs provide machine-readable routing and boundary definitions. They answer questions like:

- which module owns a kind of change
- which dependencies are allowed
- which package families are primary entrypoints

### Tasks

Task cards and milestones turn work into explicit execution units. They define scope, validation, done definition, and sequencing.

## Why Plumego separates them

One layer is rarely enough:

- docs alone are too loose for reliable automation
- specs alone are too dry for onboarding and review
- tasks alone do not explain the architecture they rely on

The repository stays understandable because these three surfaces reinforce each other without collapsing into one file or one workflow.

## What this means for contributors

Before changing code, classify the work:

1. Which stable root or `x/*` family owns it?
2. Which canonical docs describe the intended path?
3. Which spec or task card constrains the change?

That sequence reduces search radius and keeps changes small and reversible.

## What this page is not

- not a replacement for package-level docs
- not a license to skip module manifests
- not a promise that every repo rule is already stable forever

It is the working model that keeps Plumego agent-friendly and reviewable.
