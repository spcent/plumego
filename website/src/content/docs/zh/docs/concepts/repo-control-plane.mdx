# 仓库边界

Plumego 不只是一些 Go 包。它还维护了一层仓库控制平面，用来告诉人和 agent 在改代码前应该如何先完成工作分类。

如果你还没有先看 canonical 的可运行服务形态，请先去读 [参考应用](/zh/docs/reference-app)。等你的问题不再是“这个服务是怎么组装出来的”，而是“这项改动到底该归仓库里的哪一层负责”时，再回到这里。

## 什么时候这页应该接管 Reference App

当单纯对照文件布局已经不够，而你开始需要仓库级分类时，就应该从 Reference App 切到这页。

常见信号包括：

- 你在判断某个改动应该属于哪个 stable root 或 `x/*` 家族
- 你需要把服务本地布局与仓库级规则分开
- 你需要决定接下来应该以 docs、specs 还是 task cards 作为事实来源
- 你在 review 一个改动，而真正的风险已经从 runnable shape 变成边界漂移

## 三个表面

### Docs

Docs 面向人解释意图。它负责说明 canonical path、模块边界、发布姿态，以及进入仓库的默认方式。

### Specs

Specs 提供机器可读的路由和边界定义。它回答这类问题：

- 某类改动应该由哪个模块负责
- 允许哪些依赖关系
- 哪些包家族是主入口

### Tasks

Task cards 和 milestones 把工作变成显式执行单元。它们定义 scope、validation、done definition 和顺序。

## 为什么要分开

单独一层通常都不够：

- 只有 docs，对自动化来说太松
- 只有 specs，对 onboarding 和 review 来说太硬
- 只有 tasks，又解释不了它依赖的架构意图

正因为这三层彼此补位又不混成一层，Plumego 的仓库才更容易理解。

## 这对贡献者意味着什么

在改代码前，先分类：

1. 这个改动由哪个 stable root 或 `x/*` 家族负责？
2. 哪些 canonical docs 描述了预期路径？
3. 哪个 spec 或 task card 约束这次改动？

这套顺序可以减少搜索半径，也让改动保持小而可逆。

## 这页不是什么

- 不是 package 级文档的替代品
- 不是跳过 module manifest 的理由
- 也不是说所有仓库规则都已经永远稳定

它只是让 Plumego 保持 agent-friendly 和可评审的工作模型。
