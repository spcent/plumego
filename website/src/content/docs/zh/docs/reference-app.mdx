# 参考应用

`reference/standard-service` 是 Plumego 的 canonical 应用形态。当问题是文件布局、路由归属，以及哪些结构可以安全复制到真实服务时，看这页。

如果还没有运行过服务，请先看[开始使用](/zh/docs/getting-started)。如果问题已经变成归属判断，请看[仓库边界](/zh/docs/concepts/repo-control-plane)。

## 目录地图

```text
reference/standard-service/
  main.go
  internal/
    app/
      app.go
      routes.go
    config/
      config.go
    handler/
      api.go
      health.go
```

## 每个文件展示什么

| 文件 | 展示内容 | 可以复制的思路 |
| --- | --- | --- |
| `main.go` | 加载配置、构建 app、注册路由、启动服务 | bootstrap 顺序保持显式 |
| `internal/app/app.go` | 应用本地构造 | 依赖通过 constructor wiring |
| `internal/app/routes.go` | 路由注册 | 一个 method、一个 path、一个 handler 保持可见 |
| `internal/config/config.go` | 服务本地配置 | 配置解析靠近应用，不放进稳定根 |
| `internal/handler/api.go` | JSON API handler | 使用 `net/http` 签名和 `contract` 写回路径 |
| `internal/handler/health.go` | 存活/就绪 handler | 健康端点保持小而 transport-only |

## 组装图

```text
main.go
  -> config.Load()
  -> app.New(cfg)
       -> core.New()
       -> 本地 handlers 和 dependencies
  -> RegisterRoutes()
       -> /healthz
       -> /readyz
       -> /api/v1/greet
  -> Start()
```

## 可以复制什么，不应推导什么

| 可以安全复制 | 不应推导 |
| --- | --- |
| 显式 bootstrap 顺序 | 每个应用都必须有完全相同的 internal 包 |
| 应用本地 constructor 和 route 注册 | `reference` 负责仓库级边界 |
| 标准库 handler 签名 | `x/*` 包属于默认路径 |
| `contract.WriteResponse` 和 `contract.WriteError` | 鼓励为每个功能做一套响应 helper |
| 只依赖稳定根的 canonical 路径 | 已检入的实验扩展都拥有同等兼容性承诺 |

## 什么时候使用这页

| 场景 | 是否使用这页 |
| --- | --- |
| 新建 Plumego 服务 | 是 |
| 校验 handler 和 route 形态 | 是 |
| 对比服务骨架 | 是 |
| 判断能力由哪个包负责 | 否，去看[模块总览](/zh/docs/modules/overview) |
| 判断兼容性或采用姿态 | 否，去看[发布策略](/zh/docs/release-posture) |
