# 发布策略

Plumego v1 覆盖当前仓库中的已检入模块，但不同区域承载的兼容性承诺并不相同。这一页的作用，就是避免读者把“已经检入仓库”“出现在事实卡片里”或“被参考应用提到”误读成一刀切的稳定性承诺。

## 如何结合当前仓库来读发布姿态

当前仓库其实已经给出几组信号，但这些信号必须组合起来理解，而不是单看其中一条。

- 稳定根定义默认学习路径，但仍然要结合模块边界来判断
- `x/*` 家族是真实存在的仓库表面，但其中很多家族本来就比默认路径演进得更快
- `reference/standard-service` 只依赖稳定根，因此“进入 canonical 应用路径”是比“仓库里存在这个包”更强的信号
- docs、脚手架和示例可以解释如何使用，但它们不应该偷偷把某个包升级成比支持矩阵更强的兼容性承诺

## 实际判断规则

在把某个包视为长期可靠依赖之前，按顺序回答这些问题：

1. 它属于稳定根，还是属于扩展家族？
2. 支持矩阵把它标成长期默认表面，还是仍在演进的区域？
3. 它属于 canonical reference path，还是只是某个可选能力家族的一部分？
4. 目标模块的 `module.yaml` 与当前 roadmap，是否进一步收窄了该区域预期承载的改动类型？

这套顺序可以降低把实验性表面误读成长约束的风险。

## 什么时候应该放慢判断

如果出现下面任意一种情况，就不要太快把它当成稳定依赖：

- 这个包位于 `x/*` 之下
- 它没有出现在 `reference/standard-service` 中
- 这项工作明显是能力型、拓扑较重、租户相关或协议相关的问题
- 仓库文档把它当作扩展家族入口，而不是最小可运行路径的一部分

在这些情况下，更安全的默认假设是：你面对的是一个需要更谨慎核对契约的扩展表面。

## 为什么要单独写这一页

Plumego 把发布姿态写清楚，是为了避免文档、脚手架和扩展工作无意中夸大兼容性承诺。它的目的不是让仓库显得摇摆不定，而是把稳定路径收得足够窄，让读者知道哪些东西可以更放心地继承。
