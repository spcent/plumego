# Plan for M-006: v1.0.1 Maintenance Lane

Milestone: `M-006`
Objective: Keep the published `v1.0.0` state verifiable while opening a small,
reversible v1.0.x maintenance lane for generated data, CI upkeep, onboarding
truth, and post-release evidence.
Constraints: do not rewrite `v1.0.0`, do not add stable-root API, do not promote
experimental extensions without complete evidence, and commit after every
completed card.
Affected Modules: release, website, CI, docs, extension evidence.

## Phase Map

- Phase 1: Clean the current workspace drift.
- Phase 2: Harden the v1 maintenance surface.
- Phase 3: Resume extension evidence only when the ledger can be updated with
  real release evidence.

## Parallel Groups

- None. Maintenance cards are intentionally sequential so each commit can be
  reviewed and reverted independently.

## Risk Register

- Risk: generated website data drift hides real source drift.
  Mitigation: run `pnpm sync`, inspect generated diff, and commit or restore
  with evidence.
- Risk: final tag evidence becomes confusing because `main` moved after the tag.
  Mitigation: record tag-target and main-run facts without rewriting the tag.
- Risk: extension promotions are attempted too early.
  Mitigation: keep existing P2 cards blocked until release refs, snapshots, and
  owner sign-off exist.

## Verification Strategy

- Card-level checks: each card has at most three quick gates.
- Milestone-level checks: run agent workflow, module manifests, extension beta
  evidence, and `git status` before final M-006 verification.
- Remote evidence: for CI changes, use GitHub Actions run IDs rather than local
  assumptions.

## Exit Condition

- all planned cards completed or explicitly superseded
- current workspace is clean
- published v1 evidence is traceable
- extension blockers remain explicit
