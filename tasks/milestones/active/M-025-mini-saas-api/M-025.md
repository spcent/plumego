## Goal

`use-cases/mini-saas-api` exists as a runnable multi-tenant SaaS backend that
demonstrates, in one explicitly wired app, the stable roots (`core`,
`contract`, `middleware`, `security`, `store`, `health`, `log`, `metrics`)
plus the beta extensions `x/rest`, `x/tenant`, and `x/observability` — proving
Plumego can carry a real SaaS API end to end.

---

## Architecture Decisions

- New code lives only under `use-cases/mini-saas-api/` with its own `go.mod`
  and a `replace github.com/spcent/plumego => ../..` directive; the v1 build
  carries **zero external dependencies** (stdlib + Plumego only).
- Layout copies `reference/standard-service` (`main.go`, `internal/config`,
  `internal/app`, `internal/handler`, `internal/domain`); wiring stays explicit
  in `app.go`/`routes.go` — no `init()` registration, no reflection routing.
- Auth is stateless JWT via `security/jwt` (HS256, key from env) plus opaque
  rotated refresh tokens hashed into `store/kv`; passwords via
  `security/password`. No cookie sessions.
- RBAC (`owner` > `admin` > `member`) is enforced in `internal/domain/access`,
  called from handlers — never inside middleware (transport-only boundary).
- Tenant resolution, per-tenant rate limiting, and project-count quota come
  from `x/tenant` documented entrypoints only; tenant ID derives from the JWT
  claim and every repository method takes an explicit tenant ID.
- Persistence is in-memory behind repository interfaces; a SQL/SQLite backend
  is explicitly deferred (would require dependency approval, see plan §Risks).
- Idempotency for mutating endpoints uses the stable root `store/idempotency`
  (not `x/data/idempotency`), keyed by tenant + `Idempotency-Key` header.
- Success/error writes go through `contract.WriteResponse` /
  `contract.WriteError`; no app-local envelope helpers.
