# M-021: Agent-First Ecosystem Visibility

- **Branch:** `milestone/M-021-agent-first-visibility`

- **Parallel OK:** yes

---

## Goal

Make plumego's agent-first design philosophy externally visible and reusable — via a published blog-style architecture document, reusable GitHub Actions CI templates, and a `docs/AGENT_FIRST.md` guide — so other Go projects can adopt the same boundary-enforcement and machine-readable spec patterns.

---

## Architecture Decisions

- `docs/AGENT_FIRST.md` is the canonical external-facing explanation of the agent-first model: what `specs/` controls, how `internal/checks/` enforces boundaries, and how `tasks/` and `specs/change-recipes/` work together. Length: 600–900 words. No marketing language.
- GitHub Actions CI templates live in `.github/workflows/` as standalone YAML files. They must work for any plumego-based project that copies them (parameterized via inputs or env vars where needed).
- Three templates: `plumego-gates.yml` (boundary checks + vet + gofmt + tests), `plumego-benchmark.yml` (runs `internal/bench` benchmarks and comments results on PRs), `plumego-spec-validate.yml` (runs module-manifests + public-entrypoints-sync).
- The `specs/` machine-readable files are already public in the repo; this milestone adds documentation that explains their purpose to external readers and links to them from `docs/AGENT_FIRST.md`.
- Do not add any new `specs/` files or change existing spec semantics in this milestone.
- Do not add external dependencies to the workflow files (use `actions/checkout` and `actions/setup-go` only).
