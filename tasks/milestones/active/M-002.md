# M-002: Stable Roots Freeze and Reliability

- **Branch:** `milestone/M-002-stable-root`
- **Depends on:** M-001
- **Parallel OK:** no

## Goal

In 4 weeks, freeze the external behavior of stable roots and complete high-risk regression coverage so core capabilities stay durable and compatible.

## Architecture Decisions

- Do not introduce new stable public APIs; make compatibility fixes, boundary corrections, and behavior convergence only.
- Keep response and error paths on the existing canonical helpers `contract.WriteResponse` / `contract.WriteError`.
- Every change must include regression tests covering both positive and negative paths.

## Tasks

### Phase 1 — Stabilize API Surface (sequential)

1. Audit exported API inventory and mark high-priority stable symbols and behavioral contracts.
2. Map all exported API contracts and lock the list of high-priority invariants in this milestone.

### Phase 2 — Implement (parallel)

- [ ] `core`: Fix boundary regression issues in lifecycle and route registration behavior.
- [ ] `router`: Complete key regressions for static routes, path params, reverse routing, and route groups.
- [ ] `contract`: Verify consistency for error codes, response metadata, and context access paths.
- [ ] `middleware`: Lock in middleware ordering semantics and error short-circuit behavior.
- [ ] `security`: Strengthen negative tests for auth failures, signature/token failures, and timing-safe comparisons.

### Phase 3 — Regression (sequential)

1. Add minimal positive/negative tests per module and normalize test naming.
2. Align any behavior changes with existing `reference/standard-service` behavior.
3. Update change log-style notes only when behavior changes.

### Weekly checkpoints (verifiable)

1. Week 1: Complete stable-root behavior matrix (invariants and forbidden changes) and finalize change boundaries.
2. Week 2: Complete highest-priority fixes and tests for `core`, `router`, `contract`, and `security`.
3. Week 3: Complete regressions for `middleware`, `store`, `health`, `log`, and `metrics`.
4. Week 4: Run cross-module boundary gates and produce the milestone acceptance report.
