# M-001: v1 Trust Baseline

- **Branch:** `milestone/M-001-v1-trust`
- **Depends on:** (none)
- **Parallel OK:** yes

## Goal

Within 4 weeks, establish a verifiable v1 trust baseline where release status, CLI template support, and reference scaffold output are aligned, and documentation contains only executable promises.

## Architecture Decisions

- Only templates listed in both `cmd/plumego` and the docs are exposed externally.
- Only templates exposed by `cmd/plumego` and docs are presented externally.
- Publication statements in `README` and `docs/ROADMAP` must be supported by reproducible command evidence.
- CLI behavior changes must be covered by tests before documentation updates.

## Tasks

### Phase 1 — Discovery (sequential)

1. Compare actual supported templates in `cmd/plumego` against documented templates and produce a truth matrix.
2. Check release claims (version/status/tags) against reproducible repository evidence.

### Phase 2 — Implement (parallel)

- [ ] Align `new` command template allowlist with help text so supported templates match docs exactly.
- [ ] Update template entry descriptions in `cmd/plumego/README.md` and `docs/getting-started.md`.
- [ ] Rewrite release status text in `README.md`, `README_CN.md`, and `docs/ROADMAP.md` to only describe reproducible facts.

### Phase 3 — Validate (sequential)

1. Add/update CLI smoke tests: `--dry-run` success paths cover canonical/minimal/api/fullstack/microservice/canonical-like alias mapping, plus invalid-template negative cases.
2. Document or automate verification of the relationship between `reference/standard-service` and scaffold output.
3. Execute weekly check commands and submit reports.

### Weekly checkpoints (verifiable)

1. Week 1: Deliver template truth matrix and `template-audit.md`.
2. Week 2: Validate CLI allowlist and help text via `go test -run Template` (or equivalent e2e).
3. Week 3: Ensure release docs match `git` tags and facts (no unverified RC claims).
4. Week 4: Complete `reference` vs scaffold consistency verification and close the loop.

## Acceptance Criteria

```bash
go run ./internal/checks/dependency-rules
go run ./internal/checks/agent-workflow
go run ./internal/checks/reference-layout
go test -timeout 20s ./cmd/plumego/...
cd cmd/plumego && go run . new --template canonical --dry-run trust-check
cd cmd/plumego && go run . new --template rest-api --dry-run trust-check
cd cmd/plumego && go run . new --template tenant-api --dry-run trust-check
cd cmd/plumego && go run . new --template invalid-template-name --dry-run trust-check
```

Expected: all commands succeed; `invalid-template-name` returns non-zero with a clear error.

## Out of Scope

- No new template types are introduced, and no stable-root public API is expanded.
- Do not change `x/*` behavior unless it directly affects scaffold validation.
- Do not merge any documentation claims before boundary checks pass.

## Done Definition

- [ ] `gofmt -l .` returns no output.
- [ ] All acceptance commands pass, and `invalid-template-name` fails as expected.
- [ ] Documented templates match CLI behavior and repo facts.
