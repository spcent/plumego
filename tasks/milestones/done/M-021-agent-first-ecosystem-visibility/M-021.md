# M-021: Agent-First Ecosystem Visibility

- **Branch:** `milestone/M-021-agent-first-visibility`
- **Depends on:** M-019
- **Parallel OK:** yes

---

## Goal

Make plumego's agent-first design philosophy externally visible and reusable — via a published blog-style architecture document, a consolidated GitHub Actions quality gate, and a `docs/AGENT_FIRST.md` guide — so other Go projects can adopt the same boundary-enforcement and machine-readable spec patterns.

---

## Architecture Decisions

- `docs/AGENT_FIRST.md` is the canonical external-facing explanation of the agent-first model: what `specs/` controls, how `internal/checks/` enforces boundaries, and how `tasks/` and `specs/change-recipes/` work together. Length: 600–900 words. No marketing language.
- GitHub Actions CI lives in `.github/workflows/quality-gates.yml`; agent-first gates, spec validation, and optional benchmark reporting stay in one workflow entrypoint.
- The consolidated workflow runs boundary checks, vet, gofmt, tests, spec validation commands, and label-triggered benchmark reporting for PRs with the `benchmark` label.
- The `specs/` machine-readable files are already public in the repo; this milestone adds documentation that explains their purpose to external readers and links to them from `docs/AGENT_FIRST.md`.
- Do not add any new `specs/` files or change existing spec semantics in this milestone.
- Keep workflow dependencies minimal: use `actions/checkout`, `actions/setup-go`,
  and `actions/github-script` only where benchmark reporting posts PR results.
