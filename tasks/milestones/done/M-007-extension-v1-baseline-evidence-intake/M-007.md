- **Parallel OK:** no

## Goal

Turn the published `v1.0.0` tag into the first post-v1 release evidence point
for remaining experimental extension candidates without changing runtime
behavior or promoting any extension maturity status.

## Architecture Decisions

- Treat `v1.0.0` as the first usable release ref for the remaining extension
  evidence lane.
- Do not promote `x/tenant`, `x/frontend`, `x/ai` stable-tier subpackages,
  selected `x/data` surfaces, `x/discovery`, or `x/messaging` in M-007.
- A single release ref is intake evidence only. It does not satisfy the
  two-release promotion rule.
- Keep `release_history_missing`, `api_snapshot_missing`, and
  `owner_signoff_missing` blockers unless the ledger has enough evidence to
  clear the specific blocker honestly.
- Keep each card to one primary module and commit after each completed card.
